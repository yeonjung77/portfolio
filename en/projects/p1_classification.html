Summary

  • Scope : A multi-task classifier predicting 6 attributes (category, subcategory, article type, color, season, usage) for 43,005 fashion product images with a single reproducible pipeline from EDA to inference
  • Fair comparison : Trained EfficientNet-B3 and ViT-B/16 by varying only the backbone while keeping loss, optimizer, scheduler, augmentation, seed, and all other settings identical
  • Key finding : The two models differ by only 0.0002 in avg_attr_f1 despite an 8× gap in backbone capacity; the performance ceiling stems from data quality and class imbalance, not the architecture
  • Final choice : Selected EfficientNet-B3, which offers equivalent accuracy with 1/8 the parameters and meaningful gains in training and inference speed

Problem

Build a model that predicts 6 attributes from low-resolution (60×80) fashion product images. Real-world fashion retrieval and recommendation services need to extract multiple attributes per item.

Challenges identified during EDA:
  • Severe class imbalance — masterCategory ratio max/min = 21,397×
  • Long-tailed colors — 12 / 46 color classes have fewer than 50 training samples
  • Label ambiguity — humans can't reliably distinguish Beige vs Cream, Off White vs White
  • Low resolution — 60×80 pixels constrains fine color-tone discrimination

Method

6-Head Multi-task Architecture

Image [B, 3, 224, 224]
  → Backbone (timm)        → features [B, feature_dim]
  → Dropout(0.3)
  → heads["category"]      Linear → [B, 5]
  → heads["subcategory"]   Linear → [B, 30]
  → heads["articletype"]   Linear → [B, 68]
  → heads["color"]         Linear → [B, 46]
  → heads["season"]        Linear → [B, 4]
  → heads["usage"]         Linear → [B, 7]

Loss

total = 1.0 × L_category
      + 0.5 × mean(L_subcategory, L_articletype, L_color, L_season, L_usage)

All 6 heads use CrossEntropyLoss(label_smoothing=0.1).

Training setup (identical for both models)

OptimizerAdamW (backbone lr 1e-4, head lr 1e-3, weight_decay 0.01)
SchedulerCosineAnnealingLR (T_max=30, eta_min=1e-6)
Early Stoppingpatience 5, monitor avg_attr_f1 (category saturates at 99% in 1-2 epochs)
Augmentation (train)RandomResizedCrop + Flip + ColorJitter + ImageNet Normalize
Augmentation (val/test)Resize 256 → CenterCrop 224
Hyperparamsbatch_size 32, epochs 30, image 224×224, seed 42

Only the backbone differs; all other settings are held constant so that any performance difference stems purely from architecture.

Insight

Class imbalance directly determines per-class F1

The pattern is clear in the color scatter plot: all 12 classes with fewer than 50 training samples collapse to F1 ≈ 0, and the same pattern holds for usage (Party = 18, Travel = 15 training samples). When training data is sufficient the model performs reliably, but in data-scarce classes no architectural change can compensate.

  1. EfficientNet and ViT differ by only 0.0002 in avg_attr_f1. If the backbone architecture were the deciding factor, an 8× parameter increase (10.94M → 85.92M) should yield a meaningful gain.
  2. The color weakness appears identically across both backbones (0.37 vs 0.40). Architecture changes alone cannot resolve this limitation.

Confusion patterns support label ambiguity

Model Comparison — CNN vs Transformer

Hypothesis on the micro-differences

  • EffNet leads in subcategory (+0.035) — CNN's local features favor recognizing garment shape.
  • ViT leads in color (+0.030) and category_f1 (+0.020) — Transformer's global self-attention better captures holistic patterns like overall color or category.

Final choice : EfficientNet-B3

Equivalent performance, parameters at 1/8 the scale, and an advantage in both training and inference speed. Since ViT's additional capacity does not translate into improved generalization on this dataset, EfficientNet is the more practical choice.

The generalization gap is nearly identical across the two models. If the backbone influenced robustness, the ViT with 8× the capacity should have shown either greater overfitting or better generalization, yet neither pattern emerged. The main source of variance is the data — specifically the distributional instability of minority-class tasks.

Engineering notes

  • Single source of truth. 1_utils.TASK_KEYS defines the 6-task specification and 2_preprocess.COLUMN_TO_TASK handles the mapping between CSV columns and model keys. The model, loss, dataset, and trainer all reference the same source, eliminating duplicated string definitions and ensuring that typos immediately fail with a KeyError for efficient debugging.
  • Modular pipeline. Responsibilities are split across files 1–7 (utils → preprocess → dataset → model → loss → scheduler → trainer) and main.py, enabling each module to be tested independently and learning components to be swapped freely.
