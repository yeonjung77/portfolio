Summary

  • Scope : An end-to-end pipeline that takes one photo and ① detects/segments garments and parts (Stage 1), ② tags each garment's attributes (Stage 2), and ③ assigns parts (sleeves, pockets, etc.) to the right garment.
  • 2-Stage design : Stage 1 = Mask R-CNN (47-class) instance segmentation; Stage 2 = ResNet-50 multi-label (294 attributes). Guided by EDA, Stage 2 trains only on attribute-rich main garments (76,864 crops).
  • Production serving : FastAPI (REST API) + Gradio (demo UI) + HuggingFace Spaces (hosting) + GitHub Pages embed. End-to-end ~497ms, under the 500ms target.

Problem

From a full-body fashion photo, segment garments and parts at the pixel level and tag each garment's attributes (length, silhouette, material, pattern, etc.).
Detection and attribute tagging differ fundamentally in input, output, and metrics, so instead of one model I split the task into a 2-Stage pipeline: detections are cropped and passed to the attribute classifier.

Insights

1. Low part mAP is a structural limit, not a training failure

Part mAP (0.30) is half of main mAP (0.61). Parts (sleeves, collars, pockets) combine small objects + extreme rarity, making them hard to pass IoU thresholds — a structural limit already foreshadowed in EDA.

2. The macro-F1 trap — "not bad at it, just no data"

Evaluated on# attrsmacro-F1
All2940.2503
support ≥ 501060.4506
support ≥ 200550.5401 (above target)

68 attributes have zero positives in test → mathematically F1=0, yet they're included in the average. The monotonic trend proves the limit is data (rare attributes), not the model. I report both the overall and the fair metric.

3. Focal Loss can't rescue rare attributes either

BCE (0.244) vs Focal (0.245) gives near-identical F1 — Focal only shifts the precision↑/recall↓ balance. Attributes with 0–dozens of positives can't be learned by any loss. The bottleneck is data, not the loss.

Engineering Notes

  • 22× faster validation. Per-epoch validation was slower than training (75 min). Moving predicted-mask restoration from per-instance CPU to per-image GPU batch (F.interpolate) cut it to ~1.2 min.
  • Honest evaluation metrics. Overall macro-F1 alone suggests "the model is bad." Measuring fair metrics by support isolated the real cause (data).
  • Removed cold start. First inference was ~1700ms (CUDA init). A one-time warmup at server startup stabilized later requests to ~497ms.
  • Lightweight deploy. Category/attribute names were split into meta_names.json (37KB) instead of the original annotation (517MB), so the Space ships without huge files.
