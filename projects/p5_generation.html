Summary

  • 도메인 파인튜닝 : 범용 Stable Diffusion v1.5를 FashionGen으로 LoRA 파인튜닝해, 패션 전문 텍스트를 상품 이미지로 생성.
  • 비교 실험 : 파인튜닝 전 vs 후로 도메인 효과를 입증하고, LoRA rank 4 vs 8로 품질-비용 트레이드오프를 분석.
  • 정량 평가 & 배포 : 실제 캡션 1,000개로 생성해 FID 72.4 → 59.6 개선을 확인하고, 결과를 Gradio + HuggingFace Spaces 데모로 배포.

데모

※ 사전 생성된 결과입니다.

ℹ 동일한 프롬프트를 파인튜닝 전(base Stable Diffusion)과 후(LoRA) 모델로 생성한 결과입니다.
※ 막대 그래프 = CLIP 점수(생성 이미지와 텍스트 프롬프트의 일치도)
ℹ 카테고리별 프롬프트를 사용해 생성한 이미지입니다.
ℹ LoRA 강도(scale)는 학습으로 익힌 패션 스타일을 생성에 얼마나 강하게 반영할지를 뜻합니다.
0~1 사이 값이며, 0이면 파인튜닝 전 base Stable Diffusion과 같고, 1로 갈수록 FashionGen 스타일이 강하게 반영됩니다.

문제 정의

범용 Stable Diffusion은 "자카드", "맨더린 칼라" 같은 패션 전문 용어와 상품컷 스타일을 잘 표현하지 못한다. 모델을 처음부터 학습하는 것은 비현실적이므로, 기성 SD에 LoRA 어댑터만 얹어 패션 도메인에 적응시키고 그 효과를 정량적으로 검증하는 것을 목표로 삼았다.

⚠️ 프로젝트에서 마주한 어려움

검증된 라이브러리를 쓰더라도, 실제로 돌려 비교·배포하는 과정에서 여러 실무 이슈를 디버깅해야 했다.

어려움구체적 상황 / 해결
라이브러리 버전 충돌최신 transformers(5.x)·numpy(2.x)·setuptools(82)가 torch 2.1과 충돌 → 호환 버전으로 고정
정밀도(dtype) 충돌학습 중 검증 생성에서 LoRA(fp32)+베이스(fp16) 혼합으로 크래시 → autocast로 감싸 해결
평가 프로토콜FID는 소수 이미지로는 무의미 → 실제 캡션 1,000개로 대량 생성해 분포 비교
데이터 해상도 한계보유 데이터가 256px → SD 네이티브 512로 업스케일, 한계는 정직하게 문서화

데이터 & 방법

데이터셋FashionGen — 불균형(2,972배)·중복을 정리해 상위 10개 카테고리 + 상품당 1장 dedup → 49,148장 (256→512 업스케일)
캡션input_description 그대로 사용 (CLIP 77토큰 초과 10%뿐 → 자동 truncation)
모델Stable Diffusion v1.5 + LoRA (U-Net attention, VAE·Text Encoder·U-Net 본체는 frozen)
학습ε-prediction MSE · AdamW 1e-4 · 15,000 step · fp16 · 단일 GPU (diffusers 적응)

평가 방법

실제 패션 분포와 같은 조건에서 비교하기 위해, validation 캡션 1,000개로 이미지를 생성한 뒤 두 지표로 평가했다.

지표의미방향
FID생성 이미지와 실제 패션 사진의 분포 차이낮을수록 ↓ (실제에 가까움)
CLIP Score생성 이미지와 텍스트 프롬프트의 일치도높을수록 ↑

파인튜닝으로 FID가 72.4 → 59.6으로 약 16% 낮아져, 생성 결과가 실제 패션 이미지 분포에 더 가까워졌다. rank를 8로 키워도 거의 동일(오히려 미세 악화)해 rank 4로 충분(포화)함을 확인했다.

인사이트

1. 도메인 특화 효과 — 스타일 일관성

파인튜닝 전 base는 야외·라이프스타일 등 구도가 제각각인 반면, LoRA는 단색 배경 + 정면 + 모델 중앙 배치의 패션 상품컷 스타일로 일관되게 생성했다. 정성적 변화(상품컷 스타일)와 정량적 개선(FID↓)이 일치해, 도메인 적응이 실제로 작동했음을 확인했다.

2. LoRA rank에 따른 성능 포화

LoRA rank를 4 → 8로 2배 키워도 FID·CLIP이 거의 동일했다. 즉 rank 4에서 이미 성능이 포화되어, 용량을 늘릴 실익이 없었다. 과도한 용량을 가정하기보다 필요한 최소 용량을 실험으로 규명했다.

3. 정직한 한계

절대 FID(~60)는 목표(≤30)에 못 미쳤다. 원인은 데이터가 256px 업스케일이라 디테일에 한계가 있고, 평가 캡션이 치수·소재코드가 섞인 거친 실상품 설명이기 때문이다. 억지로 우회하지 않고 원인과 함께 한계로 남기고, 고해상도 데이터·캡션 정제를 향후 과제로 두었다.

엔지니어링 노트

  • 재현 가능한 파이프라인 : 데이터 전처리(H5 → imagefolder) → LoRA 학습(diffusers 적응) → 대량 생성 → FID·CLIP 평가 → 데모까지 end-to-end로 구성하고, 단계마다 검증 기준을 두었다.
  • 실무 트러블슈팅 : 버전 충돌(transformers/numpy/setuptools), 학습 중 dtype 크래시(autocast), FID 대량생성 프로토콜, 대용량 바이너리 배포(HF API 업로드) 등을 진단·해결했다.
