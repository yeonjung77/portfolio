Summary

  • 구축 대상 : 사진 1장을 받아 ① 옷·디테일을 검출/분할(Stage 1)하고 ② 각 의류의 속성을 태깅(Stage 2)한 뒤 ③ 디테일(소매·주머니 등)을 알맞은 의류에 귀속시키는 End-to-End 파이프라인.
  • 2-Stage 설계 : Stage 1 = Mask R-CNN(47-class) instance segmentation, Stage 2 = ResNet-50 multi-label(294 속성). EDA 근거로 Stage 2는 속성이 풍부한 Main 의류만(76,864 crop) 학습.
  • 프로덕션 서빙 : FastAPI(REST API) + Gradio(데모 UI) + HuggingFace Spaces(호스팅) + GitHub Pages 임베드까지. End-to-End ~497ms로 목표(500ms) 충족.

라이브 데모

사진을 업로드하면 옷·디테일 검출(파랑=의류, 주황=디테일) + 의류별 Top 속성 + 전체 JSON 트리를 실시간으로 보여준다.

문제 정의

전신 패션 사진에서 의류와 디테일을 픽셀 단위로 분할하고, 각 의류가 어떤 속성(길이·실루엣·소재·패턴 등)을 갖는지 태깅한다.
검출과 속성 태깅은 입력·출력·평가지표가 본질적으로 달라, 단일 모델 대신 2-Stage 파이프라인으로 분리했다. 검출 결과를 잘라(crop) 속성 분류기에 넘기는 구조다.

인사이트

1. 디테일(Part) mAP가 낮은 건 학습 실패가 아니라 구조적 한계

Part mAP(0.30)는 Main mAP(0.61)의 절반이다. 디테일(소매·칼라·주머니)은 작은 객체 + 극소 빈도가 겹쳐 IoU 기준을 통과하기 어렵다 — EDA에서 이미 예고된 구조적 한계다.

2. macro-F1의 함정 — "못한 게 아니라 데이터가 없는 것"

평가 대상속성 수macro-F1
전체2940.2503
support ≥ 501060.4506
support ≥ 200550.5401 (목표 초과)

test에 양성 0개인 속성이 68개 → 수학적으로 F1=0 확정인데 전체 평균에 포함된다. 빈도-F1 단조 증가가 "한계는 모델이 아니라 데이터(희귀 속성)"임을 정량 증명한다. 보고는 전체·공정 지표를 둘 다 명시했다.

3. Focal Loss로도 희귀 속성은 못 살린다

BCE(0.244) vs Focal(0.245)로 F1 거의 동일 — Focal은 precision↑ recall↓로 균형만 바꿨다. 양성이 0~수십 개뿐인 속성은 어떤 손실 함수로도 학습되지 않는다. 곁가지가 아니라 데이터가 한계다.

엔지니어링 노트

  • 검증 병목 22배 단축. 매 epoch 검증이 학습보다 느렸다(75분). 예측 마스크 복원을 인스턴스별 CPU → 이미지별 GPU 일괄(F.interpolate)로 바꿔 검증을 ~1.2분으로 줄였다.
  • 정직한 평가 지표 설계. 전체 macro-F1만 보면 "모델이 나쁘다"로 오해하기 쉽다. support(양성 개수)별 공정 지표를 함께 측정해 한계의 진짜 원인(데이터)을 분리했다.
  • cold start 제거. 첫 추론이 ~1700ms(CUDA 초기화)였다. 서버 시작 시 워밍업 추론을 1회 돌려 이후 요청을 ~497ms로 안정화했다.
  • 경량 배포. 카테고리·속성 이름을 원본 annotation(517MB) 대신 meta_names.json(37KB)으로 분리해 Space에 거대 파일 없이 배포했다.
