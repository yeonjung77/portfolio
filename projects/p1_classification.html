패션 상품 이미지 분류

Summary

  • 구축 대상 : 패션 상품 이미지 43,005장에 대해 6개 속성(카테고리, 서브카테고리, 아이템 종류, 색상, 시즌, 용도)을 단일 모델로 예측하는 멀티태스크 분류기. EDA부터 추론에 이르는 전 과정을 재현 가능한 단일 파이프라인으로 구성
  • 공정 비교 : EfficientNet-B3와 ViT-B/16을 백본만 변경하고 loss, optimizer, scheduler, augmentation, seed 등 나머지 설정은 모두 동일하게 학습
  • 핵심 발견 : 두 모델의 avg_attr_f1 차이는 0.0002로, 백본 capacity 8배 차이에도 성능이 사실상 동일. 모델 아키텍처가 아닌 데이터 품질과 클래스 불균형이 성능 한계의 주된 원인임을 확인
  • 최종 모델 선택 : 동등한 성능 대비 파라미터 1/8, 학습 및 추론 속도에서 유의미한 이점인 EfficientNet-B3를 선택

문제 정의

60×80 저해상도 패션 상품 이미지에서 6개 속성을 예측하는 모델을 구축한다. 패션 검색 및 추천 서비스에서는 상품 한 건당 다수의 속성을 추출해야 한다.

EDA에서 미리 발견한 도전 과제:
  • 극심한 클래스 불균형 — masterCategory 최대/최소 비율 21,397배
  • Long-tail color — 46개 색상 중 12개가 학습 샘플 50개 미만
  • 라벨 모호성 — Beige vs Cream, Off White vs White처럼 사람도 구분하기 어려운 쌍이 다수
  • 저해상도 한계 — 60×80 픽셀로는 미세한 색상 톤 구분이 어려움

학습 방법

6-Head 멀티태스크 구조

Image [B, 3, 224, 224]
  → Backbone (timm)        → features [B, feature_dim]
  → Dropout(0.3)
  → heads["category"]      Linear → [B, 5]
  → heads["subcategory"]   Linear → [B, 30]
  → heads["articletype"]   Linear → [B, 68]
  → heads["color"]         Linear → [B, 46]
  → heads["season"]        Linear → [B, 4]
  → heads["usage"]         Linear → [B, 7]

Loss

total = 1.0 × L_category
      + 0.5 × mean(L_subcategory, L_articletype, L_color, L_season, L_usage)

6개 head 모두 CrossEntropyLoss(label_smoothing=0.1)를 사용하였다.

학습 설정 (두 모델 동일)

OptimizerAdamW (backbone lr 1e-4, head lr 1e-3, weight_decay 0.01)
SchedulerCosineAnnealingLR (T_max=30, eta_min=1e-6)
Early Stoppingpatience 5, avg_attr_f1 기준 (category는 1~2 epoch에 99% 포화)
Augmentation (train)RandomResizedCrop + Flip + ColorJitter + ImageNet Normalize
Augmentation (val/test)Resize 256 → CenterCrop 224
공통 하이퍼파라미터batch_size 32, epochs 30, image 224×224, seed 42

백본만 다르고 나머지 모든 설정은 동일하게 하여 성능 차이가 오직 아키텍처에서 비롯되도록 설계하였다.

학습 곡선

태스크별 F1 (EfficientNet-B3)

인사이트

클래스 불균형이 클래스별 F1을 직접 결정한다

color 산점도에서 패턴이 명확히 드러난다. 학습 샘플 50개 미만인 12개 클래스가 모두 F1 ≈ 0으로 수렴하였으며, 동일한 패턴이 usage 태스크에서도 관찰된다 (Party 18개, Travel 15개 학습 샘플). 학습 데이터가 충분한 클래스에서는 모델 성능이 안정적이지만, 데이터가 부족한 클래스에서는 어떠한 아키텍처 변경으로도 보완할 수 없음을 시사한다.

  1. EfficientNet과 ViT의 avg_attr_f1 차이가 0.0002에 불과하다. 백본 아키텍처가 결정 요인이라면 파라미터 8배 증가(10.94M → 85.92M)에 따른 유의미한 성능 차이가 관측되어야 한다.
  2. color 태스크의 성능 저하가 두 백본에서 동일하게 관측된다 (0.37 vs 0.40). 아키텍처 변경만으로 해당 약점이 해소되지 않음을 확인할 수 있다.

혼동 패턴도 라벨 모호성을 뒷받침한다

모델 비교 — CNN vs Transformer

미세 차이에 대한 가설

  • EffNet이 subcategory에서 우위 (+0.035) — CNN의 local feature가 의류 형태 인식에 유리.
  • ViT가 color (+0.030), category_f1 (+0.020)에서 우위 — Transformer의 global self-attention이 색감이나 카테고리처럼 전역적 패턴에 유리.

최종 모델 선택 : EfficientNet-B3

성능은 동등하면서 파라미터는 1/8 수준이며, 학습 및 추론 속도에서도 우위에 있다. 본 데이터셋에서 ViT의 추가 capacity가 일반화 성능 향상으로 이어지지 않은 만큼, 실용성 관점에서 EfficientNet이 더 적합한 선택이다.

두 모델의 일반화 갭이 거의 동일하다. 백본이 robustness에 영향을 미친다면 capacity가 8배 큰 ViT에서 더 큰 overfitting 혹은 더 우수한 일반화가 관측되어야 하나, 두 양상 모두 나타나지 않았다. 변동의 주된 원인은 데이터, 특히 소수 클래스 태스크의 분포 불안정성에 있는 것을 확인했다.

엔지니어링 노트

  • 단일 출처 원칙(Single Source of Truth). 1_utils.TASK_KEYS가 6-태스크 명세를 정의하고, 2_preprocess.COLUMN_TO_TASK가 CSV 컬럼과 모델 키의 매핑을 담당한다. 모델·loss·dataset·trainer 모두 동일한 출처를 참조하여 문자열 중복 정의를 제거하였으며, 오타 발생 시 즉시 KeyError로 실패하도록 설계해 디버깅 효율을 확보하였다.
  • 모듈화된 파이프라인. 1~7번 파일(utils → preprocess → dataset → model → loss → scheduler → trainer)과 main.py로 책임을 분리하여, 각 모듈의 독립적 테스트가 가능하며 학습 컴포넌트의 자유로운 교체를 지원한다.
